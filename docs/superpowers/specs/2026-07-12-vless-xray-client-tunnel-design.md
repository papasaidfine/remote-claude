# VLESS/xray client tunnel for the macOS bootstrap

Date: 2026-07-12
Status: approved

## Problem

Users on poor networks (typically crossing the GFW) can reach the remote
server over plain SSH only unreliably: the connection is throttled or reset,
so VSCode Remote-SSH and `ssh remote-claude -t "claude"` stall. Wrapping that
one outbound SSH connection in an obfuscated VLESS channel (xray, usually
Reality) makes it look like ordinary HTTPS and survive the crossing.

This spec adds a client-side option to the macOS bootstrap so that connecting
to `remote-claude` automatically tunnels SSH through a local xray proxy, with
no manual step at connect time.

## Goals

- A new menu item on `local/bootstrap-macos.sh` that takes a `vless://` URL,
  installs xray, and configures a local SOCKS proxy from it.
- Connecting via VSCode Remote-SSH or `ssh remote-claude -t "claude"` routes
  SSH through xray automatically — the user does nothing beyond connecting.
- No background daemon (no launchd): xray is started on demand by the SSH
  connection itself and lingers for reuse.
- Consistent with the menu-catalog design: the item is independent,
  idempotent, and shows detected status.

## Non-goals

- **Server side is out of scope.** The VLESS server is assumed to already
  exist. This spec touches only the client (the local macOS machine). The
  chosen data path needs no configuration on the VLESS server.
- **macOS only this iteration.** Linux and Windows are deferred; they will
  reuse the same logic later. This deliberately and temporarily breaks the
  "three bootstraps are near-copies" invariant from the menu-catalog spec.
- No GUI client (no v2rayN); xray-core runs headless from its JSON config.
- No general-purpose proxy: the tunnel exists to carry SSH to the one server,
  nothing else routes through it.
- No new files committed to the repo. The runtime files (`xray.json`,
  `xray-proxy.sh`) are generated by the bootstrap in the user's home, the same
  way the script already generates the `~/.ssh/config` block.

## Approaches considered

Two forks were resolved during design:

- **Wiring: ProxyCommand + SOCKS vs. local port-forward (dokodemo-door).**
  Chosen: **ProxyCommand + SOCKS**. It lives entirely in the `~/.ssh/config`
  Host block, so VSCode Remote-SSH honors it automatically, and macOS ships the
  OpenBSD `nc` that speaks SOCKS5. (Port-forward would drop the `nc` dependency
  and unify future Windows support, but was not chosen.)
- **xray lifecycle: manual foreground vs. on-demand vs. launchd daemon.**
  Chosen: **on-demand**. The user keeps no background process running (they run
  one-shot connect commands), and the repo just removed its autostart feature,
  so a launchd daemon is off the table and a manual "keep xray running" step
  does not fit the workflow. The ProxyCommand starts xray if it is not already
  up.

## Data flow

```
VSCode Remote-SSH  /  ssh remote-claude -t "claude"
  -> ssh reads the Host remote-claude block in ~/.ssh/config
  -> ProxyCommand: ~/.config/remote-claude/xray-proxy.sh %h %p
       - ensures xray is running (starts it on demand if not)
       - nc -X 5 -x 127.0.0.1:10808 %h %p   (SOCKS5 into xray)
  -> xray VLESS outbound  ->  server's xray VLESS inbound
  -> server's freedom outbound dials %h:%p (server public IP : SSH port)
  -> server's own sshd
```

The reverse tunnel is unchanged: the same connection carries
`RemoteForward 127.0.0.1:<reverse_port> 127.0.0.1:22`, so the agent's
`ssh my-device` connect-back rides inside this SSH session exactly as before.

Because the SOCKS target is `%h:%p` — the server's **public** IP and SSH port,
taken from `HostName`/`Port` in the block — the server-side xray reaches its own
sshd via the public address (a hairpin). This needs no server configuration and
avoids VLESS servers that block private-IP destinations in routing.

## Components

### 1. New item on `local/bootstrap-macos.sh`: "xray client (VLESS URL -> local SOCKS)"

A `status_xray` / `run_xray` pair, added to the menu after the existing five
items. Idempotent and independent, like the others.

`run_xray` does four things:

1. **Prompt for a `vless://` URL** (env-var override e.g. `VLESS_URL` skips the
   prompt, matching the script's existing override pattern).
2. **Install xray-core if missing.** Prefer Homebrew (`brew install xray`) when
   `brew` is present; otherwise download the macOS release binary from
   XTLS/Xray-core into `~/.config/remote-claude/bin/xray`. Detection: an
   `xray` on `PATH`, or the vendored binary.
3. **Parse the URL and write `~/.config/remote-claude/xray.json`:**
   - inbound: SOCKS5 on `127.0.0.1:10808`, no auth, loopback only.
   - outbound: VLESS to the parsed `host:port` with the parsed `uuid`, and
     `streamSettings` built from the query parameters.
   - Supported set: `security = reality | tls | none` crossed with
     `network = tcp | ws | grpc`, honoring `flow` (e.g. `xtls-rprx-vision`),
     `sni`, `fp`, `pbk`, `sid`, `alpn`, `path`, `host`/`serviceName` as they
     apply. Parameters outside this set cause a clear error that names the
     unsupported field and writes nothing; the user can hand-edit `xray.json`.
4. **Write `~/.config/remote-claude/xray-proxy.sh`** (below) and `chmod +x` it.

`status_xray` reports done when both `xray.json` and `xray-proxy.sh` exist and
an `xray` binary is resolvable.

### 2. On-demand launcher: `~/.config/remote-claude/xray-proxy.sh`

Generated by `run_xray` via heredoc. Invoked by ssh as `xray-proxy.sh %h %p`:

```
1. If 127.0.0.1:10808 is not listening:
     nohup xray run -c ~/.config/remote-claude/xray.json \
       >~/.config/remote-claude/xray.log 2>&1 &
     poll the port until ready, timeout ~5s; on timeout, print a hint to
     check xray.log and exit non-zero.
2. exec nc -X 5 -x 127.0.0.1:10808 "$1" "$2"
```

xray, once started, stays up in the background and is reused by later
connections. No launchd, no login item.

### 3. Modification to item 4 ("Tunnel config", `write_ssh_config_block` / `run_config`)

When the xray item is configured (`status_xray` is done), `run_config` asks one
extra question: "route this tunnel through the local xray proxy? (Y/n)". If
yes, the emitted `Host remote-claude` block includes:

```
ProxyCommand ~/.config/remote-claude/xray-proxy.sh %h %p
```

If no (or xray is not configured), the line is omitted. The line is owned by
item 4 and regenerated with the whole block, so re-running item 4 never leaves
a stale ProxyCommand and never clobbers a wanted one. `status_config` reports
whether the block exists and, when it does, whether the ProxyCommand line is
present (so the menu shows proxy on/off at a glance).

An env-var override (e.g. `USE_XRAY_PROXY=1`) selects the answer for automation,
matching the existing override pattern.

## Error handling

- URL parse failure or an unsupported parameter: name the problem, write
  nothing, leave any existing `xray.json` untouched.
- xray fails to come up within the launcher timeout: ProxyCommand exits
  non-zero, so SSH reports a connection failure; the launcher's stderr points
  at `~/.config/remote-claude/xray.log`.
- `nc` absent (should not happen on macOS): `run_xray` checks for a SOCKS-capable
  `nc` up front and errors with guidance if missing.
- `~/.ssh/config` is backed up by item 4's existing mechanism
  (`*.claude-bak-<timestamp>`) before any rewrite.

## Stop / uninstall

- Stop xray (documented one-liner, no dedicated menu action):
  `pkill -f 'xray run -c .*remote-claude/xray.json'`.
- Turn the proxy off: re-run item 4 and answer "n"; the ProxyCommand line
  disappears with the block rewrite.
- Everything the item creates lives under `~/.config/remote-claude/`, so
  removal is a single directory and (optionally) a Homebrew uninstall.

## Testing

- **URL parsing:** feed a sample Reality `vless://` link and assert the key
  fields of the generated `xray.json` (uuid, address, port, security=reality,
  pbk/sid/sni/fp, flow). Repeat for a TLS+ws link.
- **Launcher idempotency:** port-already-listening path (skips start) vs. cold
  start (starts xray, waits for readiness); timeout path exits non-zero.
- **Item 4 injection:** with xray configured and "Y", the block contains the
  ProxyCommand line; with "n" it does not; re-running does not duplicate or
  strand the line.
- **End-to-end:** `ssh remote-claude 'echo ok'` succeeds through the proxy, and
  `ssh my-device 'echo ok'` (the reverse hop) still works.
- **Status:** the menu reflects real state for install / config / proxy-on.
